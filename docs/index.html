Vulkan Radix Sort

A reduce-then-scan GPU radix sort — single-header, Vulkan-only, no extra dependencies

GitHub Header README

Overview

This library sorts arrays of 32-bit unsigned-integer keys (optionally paired with 32-bit values) entirely on the GPU using Vulkan compute shaders. It implements a classic reduce-then-scan radix sort — a three-stage pipeline repeated four times, one stage per byte of the key.

Each pass extracts one byte from every key, builds a global count histogram, computes a prefix sum over that histogram, then scatters every key to its sorted position. After four passes the full 32-bit sort is complete.

Algorithm

Four Passes Over the Key Bytes

A 32-bit key is sorted one byte at a time, from least-significant to most-significant. Each pass extracts one byte as a digit in [0, 255]. Because radix sort is stable, four such passes produce a correct total order on the full 32-bit key.

Buffer Layout

All temporary data lives in a single pre-allocated storage buffer. Sizes shown for N = 4,096,000:

  • globalHistogram — four consecutive 256-entry histograms (one per pass), each accumulated from all partitions during upsweep.
  • partitionHistogram — one 256-entry histogram per partition (P = ⌈N/4096⌉), overwritten in-place by the spine with exclusive prefix sums.
  • ping-pong region — keys and values alternate between the user-supplied buffer and this scratch region each pass, so the final result lands back in the original buffer after pass 3.

N is rounded up to the next multiple of 4096 by filling spare slots with 0xFFFFFFFF. This value sorts naturally to the very end (all bits set = maximum), so the padding never corrupts the output.

Stage 1 — Upsweep (Histogram)

One workgroup per partition. 512 threads each read 8 keys (4096 total) and tally their digit into a per-partition histogram in shared memory using atomic adds. The local histogram is then flushed to global memory and accumulated into the global histogram.

Stage 2 — Spine (Prefix Sum)

256 workgroups are launched simultaneously, one per radix digit. Each workgroup scans the column of its digit across all partitions, replacing raw counts with exclusive prefix sums. The global histogram is scanned separately to produce per-digit starting offsets in the output buffer.

After the spine, partitionHistogram[p][d] holds the count of keys with digit d in all partitions before partition p. Combined with globalHistogram[d], this gives every key its exact destination slot.

Stage 3 — Downsweep (Scatter)

Each workgroup owns one 4096-element partition and writes every key (and optionally its value) to its sorted position. Within each wave (subgroup), lanes are ranked among peers sharing the same digit using a ballot bitmask — no shared-memory atomics required:

Performance

Test environment: Windows, NVIDIA GeForce RTX 5080, CUDA 13.2 / CUB v3.2.0 (Onesweep default). Median throughput at N = 2²⁵ = 33,554,432 elements, 1 warmup + 10 timed runs.

Sort type This library (Vulkan) Fuchsia (Vulkan) CUB Onesweep (CUDA)
32-bit keys only 10.66 GItems/s 13.58 GItems/s (1.27×) 22.40 GItems/s (2.10×)
32-bit key-value 9.04 GItems/s 5.02 GItems/s (0.56×) 11.68 GItems/s (1.29×)

This library outperforms Fuchsia on key-value sorts because it scatters keys and values as two separate 32-bit buffers — Fuchsia treats them as a single 64-bit key, doubling per-pass memory traffic.
